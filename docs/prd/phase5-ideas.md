# Linear Lite - Phase 5 交互模型重构与体验校准

## 背景与结论

Phase 4 的问题不在于“还不够像 Linear”，而在于当前产品仍然主要停留在一个典型 CRUD 任务系统的交互层级里。

在 2026-03-08 对 `https://linear.app/hzwjs/team/HZW/all` 的直接操作观察后，可以确认 Linear 在 Web 端的优势并不主要来自视觉皮肤，而来自以下几条更底层的设计原则：

1. **创建 Issue 是轻量起步，而不是先打开一个重型编辑器。**
2. **打开 Issue 是“阅读 + 轻编辑 + 上下文追踪”的统一界面，而不是一个表单抽屉。**
3. **List 与 Board 不是两套独立页面，而是同一视图系统的两种投影。**
4. **界面把“高频决策字段”压缩在最小操作面内，把次级信息后置。**
5. **效率来自默认值、上下文继承、键盘流和稳定的信息密度，而不是更多按钮。**

因此，Phase 5 不应继续沿着“补色值、补圆角、补阴影”的方向推进，而应进入一次 **交互模型层面的纠偏**。

本阶段允许保留 Linear 的精神，但不追求像素级复刻。目标是做出一个更适合 `linear-lite` 当前范围和复杂度的 Web 端版本。

---

## 一、Phase 4 的核心误判

Phase 4 的方向里有一部分判断是正确的，例如：

- 需要提高信息密度
- 需要减少原生控件感
- 需要让列表/看板更接近同一视觉体系
- 需要更快的新建入口和快捷键

但真正的偏差在于，它过度关注了“表面相似度”，低估了 Linear 背后的结构性设计：

### 1. 把 Issue 编辑器理解成了“更高级的 Drawer”

当前实现把 `TaskEditor` 做成了一个覆盖式右侧抽屉，仍然是典型“打开表单 -> 修改字段 -> 保存 -> 关闭”的管理后台心智。

而 Linear 的 Issue 页面不是“表单抽屉”，而是一个完整工作空间：

- 左侧或上方负责导航上下文
- 主区域负责标题、描述、评论、资源、活动流
- 右侧才是属性面板
- 状态变更和归属变更是轻量动作，不打断阅读主线

也就是说，Linear 的 Issue 不是“编辑一个数据对象”，而是“进入一个工作的上下文”。

### 2. 把新建任务理解成了“打开空白编辑器”

当前项目的新建任务会直接打开完整 `TaskEditor`，要求用户立刻面对标题、描述、状态、优先级、指派人、截止日期等一整套表单。

而 Linear 的默认新建是一个非常轻的 composer：

- 焦点先落在标题
- 描述可写但不强迫写
- 状态/优先级/Assignee/Project/Labels 都是轻量属性条
- 顶部入口和列头入口共用一套创建器，但会带上下文默认值
- 用户可以只填标题后快速创建

Linear 把“开始记录一件事”做得极轻，而不是把“完整维护一条记录”前置。

### 3. 把 List 和 Board 当成了两套组件，而不是一套视图系统

当前项目里，`TaskListView` 和 `BoardView` 的视觉与交互心智差异较大：

- List 更像表格化列表
- Board 更像轻量卡片墙
- 两者对创建、查看、状态操作、字段露出方式不一致

而 Linear 的 List / Board 是同一套视图模型的不同布局：

- 共享筛选与显示配置
- 共享分组与排序逻辑
- 共享同一 Issue 打开方式
- 共享同一创建入口逻辑
- 共享显示属性开关

### 4. 误把“像 Linear”理解为“更像设计稿”

Linear 真正强的是：

- 默认动作极少，但都很准
- 信息密度高，但层级很清楚
- hover、选中、键盘焦点、批量选择的规则非常一致
- 每个次级动作都被有意压低了视觉音量

当前项目的问题不是不够精致，而是：

- 主要动作太重
- 次要动作太显眼
- 表单感过强
- 列表与看板仍在暴露“组件”而不是“系统”

## 二、基于 Linear 真实操作的关键观察

以下观察基于直接访问 `https://linear.app/hzwjs/team/HZW/all` 后，对真实 Web 界面的操作结果。

### 1. All Issues 列表的结构特点

- 默认是高密度分组列表，而不是“表格”或“卡片列表”
- 每个分组头只保留最小信息：状态名、计数、折叠、组内新建
- 每一行 issue 的信息高度压缩，但仍有足够上下文：
  - checkbox
  - priority 图标
  - issue ID
  - 标题
  - 若干次级属性（项目、PR、assignee、创建时间等）
- 大量属性不是固定列头暴露，而是以视图配置决定显示
- “Display options” 直接控制 List/Board、Grouping、Ordering、Display properties

结论：

Linear 的列表不是“把字段排成一行”，而是“把识别任务所需的最小线索压缩进一行”。

### 2. 顶部创建 Issue 的特点

顶部 `Create new issue` 打开的是一个轻量 dialog，而非全屏页面或重型抽屉。

其特征是：

- 标题立即聚焦
- 描述区默认很轻
- 下方属性条以胶囊式轻控件呈现
- `Create issue` 是明确的主动作
- `Create more` 支持连续录入
- 用户不需要先理解全部字段，仍能快速完成新建

结论：

Linear 的新建体验追求的是 **capture speed**，不是一次填完所有数据。

### 3. 分组列头新建与顶部新建是同一个系统

从状态分组的 `+` 入口进入时，仍是同一套创建器，但默认状态已经被上下文注入，例如在 `Backlog` 组点击后默认状态即为 `Backlog`。

结论：

新建入口可以很多，但创建体验必须统一，只允许默认值不同，不允许弹出完全不同的 UI。

### 4. 打开 Issue 的结构特点

点击 issue 后进入完整 Issue 页面，核心结构是：

- 顶部 breadcrumb 与当前项导航
- 中部正文区：标题、描述、资源、活动流、评论
- 右侧属性区：状态、优先级、负责人、Labels、Project 等

这个页面的重点不是“字段编辑”，而是：

- 持续阅读
- 快速决策
- 查看上下文沉淀
- 在不中断阅读的前提下改属性

结论：

Issue 详情页应该首先是一个 **工作面板**，其次才是 **编辑表单**。

### 5. Board 不是“另一套 UI”，而是同一套视图的空间化表达

通过 `Display options` 切换到 Board 后可见：

- Board 与 List 共用同一视图配置体系
- Column/Row/Ordering/Display properties 都仍在同一个配置模型里
- 每列都保留组头、菜单、列内新建
- 卡片信息比传统 Kanban 更克制，不强调浮夸卡片块感
- Board 仍然服从信息密度和效率优先的原则

结论：

Board 的目标不是“好看”，而是当任务进入空间布局后，仍然不丢失系统性。

## 三、当前项目与 Linear 的关键差距

### 1. 新建 Issue 过重

当前项目：

- 点击 `New Issue` 后直接打开重型 `TaskEditor`
- 把所有字段一次性铺开
- 默认是“完整编辑”模式
- 保存动作偏重

Linear：

- 默认先做轻量录入
- 属性是附加配置，不是表单主角
- 允许以最少输入完成创建

问题本质：

当前实现优化的是“编辑表单”，而不是“捕获任务”。

### 2. 打开 Issue 仍是抽屉式表单，不是工作上下文

当前项目：

- `TaskEditor` 本质还是表单
- overlay 仍然存在
- 标题和描述区缺少文档式阅读氛围
- 右侧属性区与正文区关系较弱
- 没有活动流、资源、关联上下文的结构预留

Linear：

- Issue 页面是阅读主导
- 属性修改是轻动作
- 内容区、活动区、侧边属性形成稳定结构

问题本质：

当前打开任务是在“编辑数据”，不是在“进入任务现场”。

### 3. List View 仍然有较强“自制组件感”

当前项目：

- 行结构是手工拼出来的 pills 组合
- 分组头和行高虽然压缩了，但语义层级仍偏粗
- 左侧 hover 替换 priority/check 的思路是对的，但还不够稳定
- 可配置性弱，没有真正的显示属性系统

Linear：

- 行模型极度稳定
- 视图属性由统一的显示系统驱动
- 分组、排序、显示字段和布局不是散落的控件，而是统一模型

问题本质：

当前列表更像一个定制样式组件，Linear 列表更像一个视图引擎的输出。

### 4. Board 仍然过度依赖“卡片感”

当前项目：

- 卡片边框、圆角、局部块状感仍明显
- 列内内容与列头之间仍像普通看板组件
- 列的操作与列表的操作未统一

Linear：

- 卡片更薄、更克制
- 列头是强结构，卡片是低噪音内容块
- “列内新建”是主路径，不是附属按钮

问题本质：

当前 Board 还是典型 Kanban 工具的视觉延续，没有真正向效率型工作台过渡。

### 5. 顶部工具栏的信息架构过于“应用头部化”

当前项目头部：

- Logo/标题
- New Issue
- Board/List Toggle
- 搜索框
- Filter Select

这套结构虽然可用，但更像“普通业务后台”。

Linear 的上层结构更像“视图控制条”：

- 当前团队/视图
- 快速筛选与 Display
- 视图切换作为显示策略的一部分

问题本质：

当前头部优先级仍是“页面功能入口”，而不是“视图状态控制”。

## 四、Phase 5 的目标重定义

Phase 5 不再定义为 UI polish，而定义为：

> **将 Linear Lite 从“精致的任务 CRUD 页面”升级为“以视图为中心、以轻编辑为核心、以任务上下文为主导”的工作台。**

本阶段目标：

1. 重做新建任务体验，使其默认轻量、支持上下文注入。
2. 重做打开任务体验，使其从编辑抽屉升级为真正的 Issue Workspace。
3. 重组 List/Board 的视图系统，使两者共享同一配置模型。
4. 降低界面噪音，提升信息密度与操作一致性。
5. 允许视觉偏离 Linear，但必须保留其高效率、低打断、强结构的核心精神。

非目标：

- 不追求 1:1 还原 Linear 所有交互
- 不在本阶段完整实现评论/活动流后端能力
- 不在本阶段做富文本、复杂拖拽、实时协作
- 不为了“像”而硬引入超出产品复杂度的重功能

---

## 五、Phase 5 核心方案

### 方案 A：建立双层 Issue 创建模式

#### A1. 默认创建器改为轻量 Composer

新增一个专门的 `IssueComposer`：

- 默认形态为 centered modal 或 anchored floating composer
- 首屏只强调 `title`
- `description` 保持可写但降权
- `status / priority / assignee / project / due date` 以轻属性条展示
- 支持 `Create issue`
- 支持 `Create & keep open` 或 `Create more`

#### A2. 完整编辑器降级为二级路径

完整的字段编辑不应再作为默认新建入口，而应只在以下场景出现：

- 创建完成后继续完善
- 从 Issue Workspace 中进入完整编辑状态
- 需要调整较多字段时显式触发

#### A3. 统一所有新建入口

以下入口必须共用同一创建器：

- 顶部 `New Issue`
- List 分组头 `+`
- Board 列头 `+`
- 空状态创建入口
- 快捷键 `C`
- Command Palette 创建命令

不同入口仅改变默认上下文：

- 默认状态
- 默认项目
- 默认负责人或筛选继承

### 方案 B：将 TaskEditor 重构为 Issue Workspace

#### B1. 放弃“遮罩 + 表单抽屉”主模型

当前的 overlay drawer 只能作为临时过渡方案，不应继续作为长期主形态。

Phase 5 建议改为：

- `desktop`: 主区 + 右侧常驻 Issue Workspace
- 支持 route deep-link
- 左侧列表/看板保持可见，并尽量保留滚动位置
- 不再使用大面积 dim overlay

#### B2. Workspace 内部分层

Issue Workspace 拆为三层：

1. 顶部
   - ID
   - 标题
   - 快速上一个/下一个
   - 更多操作
2. 主内容区
   - 描述
   - 资源/关联
   - 未来可挂活动流、评论
3. 右侧属性区
   - 状态
   - 优先级
   - Assignee
   - Project
   - Due Date

#### B3. 阅读优先，编辑次之

标题、描述等内容区域的默认观感应更像“文档内容”，而不是“表单输入框”：

- 默认无强边框
- hover/focus 才显编辑意图
- 文本阅读状态比输入状态更重要

### 方案 C：重建统一的 View System

#### C1. View Bar 取代普通 Header Controls

顶部区域应从“页面工具栏”升级为“视图控制条”：

- 当前项目/团队上下文
- 当前 View 名称
- Filter
- Display
- Search
- New Issue

`Board / List` 不再是孤立 toggle，而属于 `Display` 或 `View Layout` 的一部分。

#### C2. 引入 Display Model

前端需要一个统一的 view state model，至少包含：

- `layout`: `list | board`
- `groupBy`
- `orderBy`
- `visibleProperties`
- `showEmptyGroups`
- `completedVisibility`

List 和 Board 都从这个模型渲染，而不是各自独立拼装。

#### C3. 统一显示属性系统

当前 List/Board 中显示哪些字段，不应写死在组件模板里。

建议把以下内容配置化：

- ID
- Status
- Priority
- Assignee
- Project
- Due Date
- Updated At
- Pull Request / Link badge

这样 List/Board 才会真正成为同一套数据投影。

### 方案 D：重新定义 List View

#### D1. List 的目标不是表格，而是“高密度任务扫描器”

List 行信息应重新排序为：

- 左侧：selection / state affordance / priority icon
- 主体：title + 次级上下文
- 右侧：时间或关键属性

#### D2. Pills 减量而不是继续堆叠

当前 pills 方向是对的，但数量和存在感仍可再降。

建议：

- 不是所有字段都做成 pill
- priority 更适合 icon，不适合总是 pill 化
- assignee 优先展示头像或缩写
- project / PR / due date 只在有值时出现

#### D3. 强化选中态与键盘导航

List 必须支持：

- 上下移动选中
- 回车打开当前 issue
- `Esc` 关闭当前 workspace
- hover、focus、selected 三种状态的明确区分

这部分是当前产品与 Linear 差距最大的地方之一。

### 方案 E：重新定义 Board View

#### E1. Board 变薄

看板卡片需要继续去“卡片组件感”：

- 更弱的边框/圆角
- 更少的视觉块感
- 更强的文本与元信息层次

#### E2. 列头更强，卡片更弱

列头应成为看板的主结构：

- 状态 icon
- 状态名
- count
- menu
- inline create

卡片只保留识别 issue 所需的最少信息。

#### E3. 列内新建做成第一等路径

当前项目列头 `+` 已有方向，但还不够像“主路径”。

Phase 5 应让列内创建具备：

- 默认状态继承
- 打开同一 composer
- 创建后可连续录入
- 不打断当前列上下文

## 六、建议的实现优先级

### P0：必须先做

1. 新建任务体验从 `TaskEditor` 分离，新增 `IssueComposer`
2. 所有创建入口统一为同一创建器
3. `TaskEditor` 重构为无遮罩或弱遮罩的 `IssueWorkspace`
4. 建立统一 `view state model`
5. 重做顶部 `View Bar`

### P1：随后跟进

1. List 行结构重排
2. Board 卡片与列头结构重排
3. Display 属性配置系统
4. 键盘导航与选中态体系

### P2：增强项

1. Issue 上一个/下一个导航
2. 资源区与活动区占位结构
3. 连续创建体验优化
4. 视图偏好记忆升级

---

## 七、Phase 5 的验收标准

### 新建 Issue

- 顶部创建、列表分组创建、看板列创建均进入同一套 `IssueComposer`
- 列表/看板上下文能注入默认状态
- 用户只输入标题也能快速创建成功
- 创建后可选择继续录入下一条

### 打开 Issue

- 打开任务后主界面仍保持工作台语义，不再是传统遮罩表单
- 标题/描述区域具有阅读优先的内容体验
- 属性修改为轻量动作，不压过正文内容

### List View

- 列表具备更高信息密度
- hover、selected、focused 状态清晰
- 支持基础键盘导航
- 字段显隐不再完全写死

### Board View

- 列头成为主要结构
- 卡片视觉权重下降
- 列内创建路径顺畅
- Board 与 List 共用显示与分组模型

---

## 八、最终判断

如果只继续做视觉微调，Linear Lite 仍会停留在“更精致的任务管理页”。

如果 Phase 5 认真完成上述调整，产品会出现一次本质变化：

- 从“表单驱动”转向“视图驱动”
- 从“编辑对象”转向“进入 issue 上下文”
- 从“两套页面”转向“同一视图系统”
- 从“模仿样式”转向“吸收 Linear 的效率结构”

这才是当前阶段真正值得投入的升级方向。
