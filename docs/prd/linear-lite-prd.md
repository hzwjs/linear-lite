# Linear Lite PRD

## 1. 文档信息

- 对应工单: `HZW-8`
- 产品名称: `linear-lite`
- 文档目标: 定义一个轻量的、受 Linear 交互模型启发的 Web 任务管理产品 MVP
- 当前优先级: 前端优先交付，使用 Vue 3 + TypeScript + Vue Router + Pinia

## 2. 背景

当前仓库尚未形成明确的产品定义与功能边界。项目目标不是完整复刻 Linear，而是先交付一个轻量、顺滑、可扩展的任务管理 Web 应用，用最小闭环验证以下核心价值：

1. 用户可以快速查看不同状态下的任务。
2. 用户可以低成本创建、编辑并推进任务。
3. 前端在后端未就绪前仍可独立开发和验证。

## 3. 产品定位

`linear-lite` 是一个面向小团队或个人项目的轻量任务管理工具，强调：

- 快速进入任务上下文
- 少层级、少打断的编辑体验
- 明确的状态流转
- 为未来接入真实后端保留稳定的数据与服务边界

## 4. 目标用户

### 4.1 核心用户

- 需要管理日常任务的小团队成员
- 希望用更轻方式替代表格/便签的个人开发者
- 需要一个可演进样板来承载前后端联调的项目团队

### 4.2 典型场景

- 每天打开应用，快速看到哪些任务待开始、进行中、已完成
- 新增一条任务并补充描述、优先级
- 处理过程中修改任务内容并推进状态
- 用关键字或状态快速定位目标任务

## 5. 目标与非目标

### 5.1 产品目标

- 提供一个可用的任务主视图，按状态浏览任务
- 支持任务创建、编辑、状态变更、优先级设置
- 支持基础搜索与筛选
- 保持前端架构清晰，后续可平滑迁移到 Spring Boot + MySQL

### 5.2 非目标

- 不做完整的 Linear 克隆
- 不做复杂权限体系、团队空间、多项目管理
- 不做评论、附件、提醒、通知、自动化规则
- 不在本期实现真实后端接口或数据库持久化
- 不默认纳入拖拽排序、实时协作、富文本编辑

## 6. 产品原则

- 轻量优先: 先做完成任务闭环所需的最小功能
- 速度优先: 常用动作尽量在同一视图内完成
- 清晰优先: 状态、优先级、搜索条件应一眼可见
- 可迁移优先: 前端调用 mock service，但接口定义应接近未来真实 API

## 7. MVP 范围

### 7.1 In Scope

- 单一任务工作区首页
- 按状态分组查看任务
- 创建任务
- 编辑任务标题、描述、优先级、状态
- 快速状态流转操作
- 基础搜索
- 基础状态/优先级筛选
- 基于 Pinia 的前端状态管理
- 基于 typed service adapter 的 mock 数据层

### 7.2 Out of Scope

- 用户登录/注册
- 多 workspace / 多团队切换
- 任务评论、附件、子任务
- 自定义字段、自定义工作流
- 后端接口、数据库设计落地
- 移动端独立设计稿

## 8. 核心实体定义

### 8.1 Task

任务对象至少包含以下字段：

- `id`: 唯一标识
- `title`: 标题，必填
- `description`: 描述，选填
- `status`: 状态
- `priority`: 优先级
- `createdAt`: 创建时间
- `updatedAt`: 更新时间

### 8.2 Status

MVP 默认状态枚举：

- `todo`
- `in_progress`
- `done`

说明：

- 为保持 MVP 简洁，首版不引入更多工作流状态。
- 如后续需要 `backlog` 或 `canceled`，通过新增 issue 扩展，不在本期隐式扩 scope。

### 8.3 Priority

MVP 默认优先级枚举：

- `low`
- `medium`
- `high`
- `urgent`

## 9. 信息架构

### 9.1 页面结构

MVP 建议采用单页主工作台结构：

- 顶部: 产品标题、搜索框、筛选入口、创建按钮
- 主体: 按状态分组的任务列表
- 右侧抽屉或弹层: 任务详情与编辑表单

### 9.2 路由建议

- `/`: 任务首页
- `/tasks/:taskId`: 打开指定任务详情，支持深链接

## 10. 核心用户流程

### 10.1 浏览任务

1. 用户进入首页
2. 默认看到按 `todo`、`in_progress`、`done` 分组的任务列表
3. 每条任务至少展示标题、优先级、状态

### 10.2 创建任务

1. 用户点击“新建任务”
2. 打开表单或抽屉
3. 填写标题、描述、优先级、初始状态
4. 提交后任务立即出现在对应状态分组中

### 10.3 编辑任务

1. 用户点击任意任务
2. 进入详情抽屉或详情页
3. 修改标题、描述、优先级、状态
4. 保存后主列表同步更新

### 10.4 快速状态流转

1. 用户在任务卡片或详情区域触发“移动到下一状态”或直接选择目标状态
2. 系统立即更新任务状态
3. 任务从原分组移动到目标分组

说明：

- MVP 采用显式状态操作作为必做项。
- 拖拽交互可作为后续增强，不作为首版依赖。

### 10.5 搜索与筛选

1. 用户输入关键字搜索任务标题
2. 用户可按状态或优先级筛选
3. 列表即时反馈结果，未命中时给出空状态提示

## 11. 功能需求

### 11.1 任务列表

- 必须按状态展示任务
- 默认按更新时间倒序展示
- 空状态需要给出清晰提示和创建入口

### 11.2 任务创建与编辑

- 标题必填，空值不可提交
- 描述支持纯文本输入
- 状态与优先级通过明确控件选择
- 编辑后应立即反映到当前视图

### 11.3 搜索与筛选

- 搜索至少匹配标题
- 状态筛选支持单选或多选，优先实现单选
- 优先级筛选支持单选
- 筛选条件变化后无需整页刷新

### 11.4 反馈与异常

- 表单校验失败时给出明确提示
- mock service 失败时提供基础错误提示与重试入口
- 列表、详情、空状态需有一致的加载/空数据反馈规范

## 12. 技术与架构要求

### 12.1 前端

- 使用 Vue 3 + TypeScript + Composition API（`<script setup>`）
- 使用 Vue Router 管理页面与深链接
- 使用 Pinia 管理任务列表、筛选条件、当前选中任务等状态
- 视图层不直接耦合 mock 数据细节

### 12.2 域模型

前端共享 typed domain model：

- `Task`
- `Status`
- `Priority`

要求：

- UI 层与 service adapter 共享同一套类型定义
- 类型命名与字段设计需便于未来映射到 Spring Boot DTO

### 12.3 服务边界

必须定义独立任务服务接口，例如：

- `listTasks`
- `getTask`
- `createTask`
- `updateTask`
- `transitionTask`

要求：

- 当前可由内存/mock adapter 实现
- 未来替换为 Spring Boot HTTP API 时，不应迫使 UI 重写核心流程

### 12.4 后端对齐约束

- 本期不实现后端
- 但接口契约应能自然映射到未来 Spring Boot + MySQL 实现
- 字段命名保持稳定，避免前端先使用难以迁移的临时结构

## 13. UI/交互要求

- 风格参考 Linear，但仅借鉴其“快、轻、少打断”的交互思路
- 首页应优先保证桌面端体验，同时具备基础移动端可用性
- 创建/编辑流程应尽量在当前上下文完成，减少页面跳转
- 重要操作按钮和状态标签应具备清晰视觉层级

## 14. 验收标准

### 14.1 文档验收

- PRD 明确描述产品目标、范围、核心流程、实体模型与技术约束
- 文档内容与项目 baseline 一致，不引入未说明的大范围功能
- 文档能为前端 Vue 3 MVP 实现提供直接指导

### 14.2 UI Walkthrough 验收路径

以下路径必须能被未来实现直接覆盖：

1. 打开应用首页，看到按状态分组的任务列表
2. 输入关键字筛选出目标任务
3. 打开任务详情并修改标题或描述
4. 调整优先级与状态并保存
5. 返回主列表后，在对应状态分组中看到更新后的任务
6. 新建一条任务，并确认其出现在所选状态下

## 15. 发布分期建议

### Phase 1: Frontend MVP

- 完成任务主视图
- 完成任务创建/编辑/状态流转
- 完成基础搜索与筛选
- 使用 mock service 跑通完整演示闭环

### Phase 2: Backend Integration

- 用 Spring Boot + MySQL 提供真实任务接口
- 替换 mock adapter
- 保持前端页面与 store 调用方式尽量不变

### Phase 3: Follow-up Enhancements

- 拖拽改状态
- 更多工作流状态
- 评论/活动记录
- 更强筛选能力

说明：

- Phase 3 的内容均应拆分为单独 Linear issue，不并入当前 ticket。

## 16. 假设与开放问题

### 16.1 当前假设

- 首版面向单团队/单工作区使用
- 单语言交付即可，不要求国际化
- 搜索只覆盖标题已能满足 MVP
- 状态流转以显式选择为主，优先级高于拖拽体验

### 16.2 开放问题

- 是否需要在 MVP 中同时提供列表视图与看板视图切换
- 是否需要在任务卡片中直接编辑优先级
- 后续是否要引入 `backlog` 状态

以上开放问题不阻塞当前 PRD 生效，但若进入实现阶段，应通过新增 issue 明确决定。
