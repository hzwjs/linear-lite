# Linear Lite - Phase 2 Team-Ready Foundation

## 定位声明

Phase 1 (MVP) 已经成功证明了轻量、单体任务交互流程的可行性。
Phase 2 的目标不再是“继续增强单页 Issue 管理”，而是迎来一次产品层级的跃迁：**达到团队真实日常使用的最低要求。**

从“个人/轻团队可用的任务板”升级为“团队真实可用的协作型任务系统”，必须首先补齐让多人能够在同一套平台上共享与隔离任务的基础设施。

---

## 阶段规划

为了确保在真实环境里的持续可用性，我们将后续演进重构为新的三个阶段。这三者层层递进，每一阶段都带来新的团队协作闭环。

### Phase 2: Team-Ready Foundation (团队可用最小闭环)

**目标**：把单机 Issue 板升级为团队可用系统。

**范围 (P0 必选基础)**：

1. **[Backend] 真实后端与持久化**：使用 Spring Boot + MySQL 替换 Mock Service，允许数据多端共享。
2. **[Users] 基础鉴权与成员分配**：引入最简登录态与用户系统，允许明确任务的“创建者 (Creator)”和“负责人 (Assignee)”。
3. **[Projects] 项目隔离空间**：增加项目维度概念，使不同项目组的任务相互隔离，避免任务池失控。

**完成标准**：

- 多个成员可同时访问同一系统。
- 前端全面脱离 `localStorage` 依赖，请求真实接口。
- 每个任务都有明确的归属（属于哪个项目、由谁创建、分配给谁）。
- 在 UI 上，不同项目的任务必须能在不同的板块下互不干扰地流转。

---

### Phase 3: Collaboration Context (协作信息沉淀)

**目标**：让协作信息沉淀在任务内，而不再只依赖外部IM工具。

**范围 (P1 协作增强)**：

1. **[Comments] 内部评论**：允许在任务下方发帖讨论上下文。
2. **[Activity] 活动流水日志**：自动记录任务的状态、优先级等流转变更记录以便追溯。
3. **[Hooks] 基础通知或 Webhook 点**：为外挂 IM (如钉钉/飞书) 留出消息推口。

**完成标准**：

- 过去的信息更改能被清楚看到“谁在什么时候改了什么”。
- 围绕某个 Bug 或任务的所有争论与结论，都可以直接挂载在 Issue 里面被新人搜索到。

---

### Phase 4: Planning & Delivery Rhythm (规划与交付节奏管理)

**目标**：支持团队按敏捷节拍组织交付，而不只是无情地消耗任务池。

**范围 (P2 节奏管理)**：

1. **[Cycles/Sprints] 迭代周期**：引入明确的开始/结束时间点。
2. **[Backlog] 待办排序**：提供单独的 Backlog 视图以便规划近期目标。
3. **[Views] 更强视图**：支持按迭代、按负责人多维度的交叉筛选与视图保存。

**完成标准**：

- 团队能直观地看到“当前迭代”正在做的事与下个迭代的目标隔离。

---

## 路线执行摘要

基于“团队最小可用性”判定，我们将沿 **单一的演进主轴** 进行：

> `Backend` ➔ `Users & Assignees` ➔ `Projects`

此阶段核心目标就是落地这 3 个闭环，缺一不可。Comments（Phase 3）和 Cycles（Phase 4）将在核心架构稳定后作为后续迭代引入。
