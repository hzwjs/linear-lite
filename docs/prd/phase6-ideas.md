# Linear Lite - Phase 6 Linear 化改造清单

## 背景

基于 2026-03-08 对本地运行中的 `linear-lite` 的实际走查，本项目当前已经具备了 Linear 式产品的基础骨架：

- 左侧项目导航
- 中部 Board / List 视图切换
- 顶部搜索、筛选、分组、排序
- Issue 打开后的独立 workspace / drawer

问题已经不再是“缺少模块”，而是这些模块组合起来后，整体体验仍更接近一个轻量 CRUD 后台，而不是一个高密度、克制、连续、可高速处理 issue 的工作台。

因此，Phase 6 的目标不应继续停留在局部润色，而应进行一次明确的 **Linear 化收敛**：

1. 统一视觉 token，建立更冷静、更精密的基础界面语言。
2. 收紧信息密度，让列表、看板、详情页都进入“高效扫描”状态。
3. 重新梳理主次层级，降低控件噪音，强化内容优先。
4. 把 Issue 详情从“表单编辑器”继续推向“持续工作的上下文界面”。

本阶段不追求像素级复刻 Linear，但要求用户在首次进入产品时，明确感到：

- 这是一个 issue workflow 工具，而不是任务管理后台
- 这是一个能高频浏览、筛选、打开、决策的界面，而不是一个堆控件的页面
- 打开 issue 后是在进入工作现场，而不是在填写表单

---

## 一、当前产品与 Linear 的核心差距

### 1. 整体视觉气质仍偏“通用 SaaS 后台”

当前界面的问题不是丑，而是不够“准”：

- 大面积浅灰背景 + 白块容器 + 明显描边的组合仍然偏后台模板化
- 蓝紫色按钮与高亮态存在感偏强，容易把注意力拉向控件而非内容
- 圆角偏大，按钮和输入框有轻微“温和 SaaS”质感，不够克制
- 文字层级、线条层级、表面层级之间还没有形成足够稳定的秩序

Linear 的视觉并不花哨，但有一种非常明确的系统感：

- 大多数区域都很轻
- 强调色只在真正关键处出现
- 大部分组件的存在感都低于内容本身
- 用户会先看到 issue，再看到状态和操作

### 2. 信息密度偏低，导致工作效率感不足

当前 Board 和 List 都有类似问题：

- 容器面积较大，但有效信息较少
- 行高、卡片 padding、顶部工具栏高度仍然偏松
- 元信息展示还不够紧凑，导致扫读速度偏慢
- 空白很多，但不是经过控制的留白，而是信息装载率偏低

Linear 的核心优势之一就是：**紧凑，但不挤；密集，但不乱。**

### 3. 主次层级不够清楚

当前页面中很多东西都在发声：

- `New issue`
- Board / List 切换
- 搜索框
- 多个筛选下拉
- Display chips
- 列边框、卡片边框、详情侧栏边框

这些元素 individually 都合理，但组合后会造成一个问题：

用户没有一个特别清晰的默认阅读路径。

Linear 的页面通常会隐式告诉用户：

1. 先看内容
2. 再看状态和归属
3. 最后才看筛选、设置和次级动作

当前产品在这个顺序上还不够稳定。

### 4. Issue workspace 仍带有明显“编辑表单”心智

当前 drawer / workspace 已经比典型弹窗更进一步，但仍有几个明显问题：

- 标题区与正文区过空
- 右侧 details 很像普通管理后台表单面板
- 底部 `Back / Save` 明显强化了“表单提交”感
- 中部 Context 模块更像占位说明，而非真实的工作上下文

Linear 的 issue 详情首先是一个工作空间，其次才是一个可编辑对象。用户进入后应该优先感到：

- 我在阅读一个 issue
- 我在处理上下文
- 我可以顺手修改属性

而不是：

- 我打开了一条数据
- 我正在填一张表

## 二、Phase 6 的设计目标

### 1. 建立更接近 Linear 的基础界面语言

目标不是“换一套颜色”，而是统一以下系统要素：

- 灰阶层次
- 文字层级
- 圆角尺度
- 边框强度
- hover / selected / focus / active 的状态语言
- 主要动作与次要动作的音量差

验收标准：

- 整个页面看起来像一个系统，而不是一组独立组件
- 即使去掉品牌色，页面也仍然成立
- 大部分控件都比 issue 内容更安静

### 2. 把 List 做成真正的高密度主工作面

List 是最能决定“像不像 Linear”的视图。

Phase 6 需要把当前的 List 从“分组后的简单列表”推进为：

- 可高速扫读
- 行结构稳定
- 字段对齐清楚
- hover / selected / keyboard focus 明确
- 分组、排序、显示属性真正形成统一系统

验收标准：

- 用户可以快速浏览多条 issue，不会被空白和控件打断
- 行级交互反馈清晰，但不喧宾夺主
- 选中某一行后，workspace 与列表之间的关系自然顺畅

### 3. 把 Board 收敛成“同一视图系统的另一种投影”

Board 现在已经有基础能力，但还残留较强的“卡片组件思维”。

Phase 6 应继续推进：

- 卡片更薄
- 列结构更轻
- 卡片信息更紧
- 视觉逻辑与 List 保持同源

目标不是把 Board 做得更花，而是让它与 List 一样高效，只是更适合空间布局和状态分布阅读。

### 4. 让 Issue drawer 更像工作现场

Issue workspace 需要从“可编辑”升级到“可持续工作”：

- 标题、描述、上下文、活动、资源之间要形成更像文档的阅读结构
- 右侧属性区应更轻，不像表单集合
- 顶部导航与 issue 身份信息要更稳定
- 底部操作不应继续强化一次性提交心智

## 三、Phase 6 改造清单

### A. Design Tokens 与全局基础样式

这是最优先的地基工作。

需要调整的方向：

- 把当前灰阶进一步拉开层次，但降低“灰底 + 白块”的模板感
- 减弱多数边框的存在感，保留必要的结构线
- 收紧圆角体系，避免按钮、输入框、面板看起来过软
- 降低主色使用频率，让高亮只出现在真正的关键动作与选中态
- 统一按钮、输入、select trigger、chip、row 的状态反馈节奏

建议结果：

- 页面更冷静
- 控件更轻
- 内容更突出

### B. 顶部工具栏重构

当前工具栏最大的问题不是功能多，而是“每个控件都像一个完整表单控件”。

Phase 6 要做的不是删功能，而是压缩表达：

- 统一搜索框、筛选器、视图切换器的尺寸和视觉强度
- 降低下拉 trigger 的表单感
- 重新组织 `Display` 区域，避免在主工作区上方产生第二层工具噪音
- 明确主按钮、次级筛选、显示配置之间的层级

预期结果：

- 工具栏像命令带，而不是筛选表单行
- 用户不会在一进入页面时先被控件吸走注意力

### C. List View 重做

这是 Phase 6 的核心任务。

需要明确推进以下方向：

1. 行模型重做
   - 行高收紧
   - issue 标题作为绝对主信息
   - 状态、优先级、负责人、日期等辅助信息重新压缩到稳定区域
   - 减少“胶囊堆叠感”

2. 分组头重做
   - 更像结构标题，不像一块列表卡片
   - 折叠、计数、组内创建保持低噪音但可发现

3. 交互状态重做
   - hover
   - selected
   - keyboard focus
   - active row
   这些状态要形成连续语言，而不是彼此各自独立

4. workspace 联动增强
   - 当前行与右侧详情建立稳定映射
   - 当 issue 打开时，列表仍保持作为导航母体存在

目标结果：

- List 成为真正的默认工作界面
- 用户可以在这里快速扫读、定位、进入 issue

### D. Board View 收敛

Board 不是展示型看板，而是任务分布视图。

需要继续压缩：

- 列头高度
- 列间视觉分隔方式
- 卡片内信息的占用面积
- 卡片 hover / selected 态

同时保证和 List 的统一性：

- 相同字段使用相同语义
- 相同状态使用相同图形语言
- 相同 issue 打开方式使用相同反馈

目标结果：

- Board 不再像单独设计的一套组件
- Board 和 List 看起来属于同一个产品系统

### E. Issue Workspace / Drawer 重构

这是第二优先级核心任务，仅次于 List。

重点不在新增多少信息块，而在于重塑界面心智：

1. 顶部结构
   - issue key、标题、前后切换、关闭动作的关系要更稳定
   - 顶部不应像一个普通 modal header

2. 主内容区
   - 标题与描述形成文档阅读流
   - Context 模块从“说明占位”升级为真实信息结构预留
   - 资源 / 活动 / 评论等区域要准备好未来演进空间

3. 右侧 details
   - 更轻、更紧、更少表单感
   - 属性编辑像轻操作，不像填写字段

4. 底部动作
   - 重新审视 `Back / Save`
   - 如果数据是局部即时保存，就不应保留传统表单提交语言

目标结果：

- 打开 issue 后，用户感受到的是工作上下文，而不是编辑弹窗

## 四、推荐执行顺序

### Phase 6.1: 先做视觉地基

- Design tokens
- 全局按钮 / 输入 / dropdown trigger 统一
- 工具栏视觉收敛

理由：

如果不先统一底层语言，后面重做 List 和 Drawer 时会不断返工。

### Phase 6.2: 优先重做 List

- 行模型
- 分组头
- 选中态
- workspace 联动

理由：

List 最能决定产品气质，也最能提升“像 Linear”的第一感知。

### Phase 6.3: 收敛 Board

- 列头、卡片、信息密度
- 和 List 的字段语义统一

理由：

Board 应该跟着 List 的语言走，而不是平行再设计一套。

### Phase 6.4: 重构 Issue workspace

- 顶部结构
- 主内容阅读流
- details 区轻量化
- 底部动作策略调整

理由：

当 List 与 Board 已经进入正确的阅读密度后，workspace 的问题会更容易看清，也更容易落地。

## 五、Phase 6 的验收原则

本阶段不以“加了多少功能”为主要衡量标准，而以以下问题是否被解决为标准：

### 1. 首屏是否更像专业 issue 工作台

用户打开页面后，不应首先感到：

- 表单很多
- 筛选器很多
- 一堆卡片浮在页面上

而应首先感到：

- 这是一个高密度、高效率的工作界面

### 2. List 是否足够快

用户能否在短时间内：

- 扫过多条 issue
- 找到目标项
- 打开目标项
- 回到列表继续处理下一条

### 3. Board 是否仍保持系统感

切到 Board 后，用户不应觉得进入了另一套产品，而应感觉只是同一信息的另一种布局。

### 4. Issue 打开后是否更像“进入上下文”

用户打开一条 issue 后，是否会自然地继续阅读、判断、编辑，而不是觉得自己进入了一个待提交表单。

---

## 六、结论

Phase 6 的重点不是继续“补一些更像 Linear 的细节”，而是把已经具备雏形的产品骨架，真正收敛成一套有明确审美与交互纪律的工作系统。

如果说前几个阶段主要是在补齐能力，那么 Phase 6 要做的是：

- 把界面从“有功能”推进到“有秩序”
- 把体验从“能用”推进到“高频使用不累”
- 把 Issue 从“数据对象”推进到“工作对象”

这才是当前 `linear-lite` 和 Linear 之间最真实、也最值得投入的差距。
