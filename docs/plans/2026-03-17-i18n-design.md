# Internationalization Design

## Overview

为 Linear Lite 增加前端国际化能力，支持中文与英文界面切换。首期覆盖前端 UI 文案、前端本地产生的错误提示、以及状态/优先级等固定枚举展示文案；不改后端接口异常的国际化体系。

## Goals

- 用户可在应用内切换 `zh-CN` / `en`。
- 刷新页面后保留语言选择。
- 核心主流程界面随语言切换而更新。
- 状态、优先级、固定按钮/标题、前端校验提示支持双语。
- 语言切换不影响现有前后端业务逻辑与数据结构。

## Scope

- 本期纳入：
  - 顶层壳与导航
  - 登录/注册页
  - Board/List 主工作区
  - Issue Composer、Task Editor
  - Project Settings、Create Project、Task Import、Attachments 等弹层/区块
  - 前端本地产生的错误与校验提示
  - 状态、优先级、固定 action label
- 本期不纳入：
  - 后端直接返回的异常 `message` 体系
  - 用户输入的富文本正文内容
  - README / docs / 测试名称文本本身

## 1. Architecture

### 1.1 Library Choice

- 使用 `vue-i18n` 作为唯一国际化方案。
- 不自建文案字典运行时，避免重复造轮子。
- 不改变现有业务 code，例如 `status: 'in_progress'`、`priority: 'urgent'`，仅对展示层做多语言映射。

### 1.2 New Modules

- `src/i18n/index.ts`
  - 创建并导出 i18n 实例
  - 注册 `zh-CN` 与 `en`
  - 配置默认语言与 fallback 语言
- `src/i18n/messages/zh-CN.ts`
- `src/i18n/messages/en.ts`
  - 按功能域维护文案
  - 推荐分组：`app`、`auth`、`task`、`project`、`import`、`attachments`、`errors`、`status`、`priority`
- `src/store/localeStore.ts`
  - 保存当前语言
  - 提供切换方法
  - 持久化到 `localStorage`
- `src/utils/i18n.ts` 或等价薄封装
  - 给组件外逻辑使用 translator
  - 支持工具函数、枚举 label helper、少量 store 内提示

### 1.3 Design Principle

- 国际化只影响展示，不影响业务数据。
- API 请求、路由参数、Pinia store 内部状态、筛选和排序仍使用稳定的英文 code。
- 文案 key 按功能域组织，不按单个组件文件组织，降低后续重构成本。

## 2. Language Switching And Data Flow

### 2.1 Entry Point

- 将语言切换入口放在全局稳定区域，优先选择侧栏底部用户区域附近或其菜单中。
- 不新增独立设置中心，避免为单一配置项扩展过度 UI。

### 2.2 Locale Resolution

- 应用启动时按以下顺序确定语言：
  1. `localStorage` 中已保存的用户选择
  2. 浏览器语言
  3. 默认值 `zh-CN`
- 浏览器语言中，中文环境映射为 `zh-CN`，其他默认 `en` 或回退到 `zh-CN`。本设计建议：
  - 首次启动按浏览器推断
  - 最终 fallback 使用 `zh-CN`

### 2.3 Runtime Flow

- 在 `src/main.ts` 中先初始化 i18n，再挂载 Vue app。
- `localeStore` 负责：
  - 读取初始 locale
  - 执行切换
  - 将最新值同步回 `i18n.global.locale.value`
- 组件内统一使用 `t()` / `$t()` 读取文案。
- 组件外逻辑通过共享 translator 入口读取文案，不把 `useI18n()` 硬塞进非组件模块。

### 2.4 Fallback Rules

- key 缺失时回退到 fallback 语言，不渲染空字符串。
- 开发阶段保留 missing key 警告，便于补齐翻译。
- 枚举文案映射必须有兜底，未知值显示原始 code，而非空白或报错。

## 3. Component Migration Strategy

### 3.1 Priority Order

按“全局外壳优先、核心路径优先、边缘功能后补”推进：

1. `src/App.vue`
2. `src/views/LoginView.vue`
3. `src/views/BoardView.vue`
4. `src/components/IssueComposer.vue`
5. `src/components/TaskEditor.vue`
6. 项目设置、创建项目、导入、附件相关弹层与区块

### 3.2 Text Categories

- 直接模板文案：按钮、标题、空状态、tooltip、占位符
- 前端生成提示：校验失败、兜底错误、操作反馈
- 枚举展示文案：状态、优先级、活动动作等

### 3.3 Backend Message Handling

首期不改后端异常国际化，因此前端采用包装策略：

- 已知错误场景：
  - 优先映射到本地 i18n key
- 未知错误场景：
  - 保留后端原始 `message`

这样可以先完成前端国际化能力建设，同时不阻塞现有后端行为。

## 4. Testing Strategy

### 4.1 Unit Tests

- 为状态/优先级 label helper 增加双语断言
- 为组件外 translator helper 增加基础单测
- 为 locale store 增加初始化、切换、持久化测试

### 4.2 Component Tests

- 必须断言文案时，在测试内显式设置 locale
- 优先断言语义结果与交互行为，避免把测试写成脆弱的整段文案快照

### 4.3 Integration Coverage

- 增加一个最小集成测试：
  - 初始渲染
  - 切换语言
  - 核心界面文案变化
  - 业务操作仍正常

---

## 5. Acceptance Criteria

- 用户可在 UI 中切换中文与英文
- 刷新页面后仍保持所选语言
- 核心主流程界面没有明显硬编码残留
- 状态、优先级、固定按钮/标题文案会随语言切换
- 前端本地错误提示可双语显示
- 切换语言不影响任务加载、创建、编辑、筛选等业务行为

---

## Risks And Non-Goals

- 风险：现有文案硬编码分散，首次替换容易漏项
- 风险：部分测试可能直接断言旧文案，需要同步重构
- 风险：后端 message 中英混杂，首期只能通过前端包装覆盖部分已知场景
- 非目标：本期不做后端异常国际化、不做多语种扩展框架以外的文档翻译
