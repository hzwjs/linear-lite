# 任务详情评审结论与优化方向设计说明

**日期**: 2026-04-10  
**状态**: 已定稿（基于 `/Users/huangzhiwen/Downloads/task_detail_complete_final.html` 与当前仓库真实实现联合评审）

---

## 1. 背景与目标

- **背景**: 用户提供了一份任务详情页评审稿，指出进度控件、逾期提示、评论与活动信噪比、附件空态、右侧属性面板等问题。
- **评审基准**: 不是只看静态稿本身，而是结合当前仓库真实实现判断哪些结论仍成立、哪些已被现有能力覆盖、哪些建议会与当前产品语义冲突。
- **目标**: 产出一份工程可落地的评审定稿，避免把静态稿问题直接等同于线上产品语义问题，并为后续实现提供清晰优先级。

---

## 2. 评审结论

### 2.1 明确成立的结论

- **活动流淹没有效沟通**：成立。当前评论与活动虽已拆成两个 section，但活动区仍按逐条系统日志展示，缺少合并与降噪。
- **附件空态占用纵向空间**：成立。无附件时仍保留完整 section 与空文案，信息密度偏低。
- **右侧属性面板扫描成本高**：成立。当前 status / priority / assignee / planned start / due date / progress / labels / project 线性堆叠。
- **翻页控件缺少来源上下文**：成立。当前仅展示 `position / total` 与左右切换，没有说明处于哪个列表结果中。
- **子任务协作信息偏弱**：基本成立。当前子任务区已有状态与标题，但缺少 assignee / due date 等执行维度。

### 2.2 部分成立、但原结论表述过强

- **进度 slider 是根本性错误**：仅部分成立。当前 `progressPercent` 是真实领域字段，并且与状态存在联动；问题不在于“绝不能手动维护”，而在于任务存在子任务时，手动进度与子任务完成度可能冲突，且 UI 没有解释两者关系。
- **评论输入区功能薄弱**：仅部分成立。当前评论区已接入 `TiptapEditor` 与 `@` mention，不再是纯文本输入框；但附件、快捷键提示、提醒成员交互仍可增强。
- **子任务圆圈按钮与 checkbox 预期错配**：仅部分成立。当前是状态选择控件而非 checkbox；真实问题是状态语义可见性不足，而不是控件语义完全错误。

### 2.3 需要改写或删除的结论

- **“进度应自动计算 = 已完成子任务 / 总数”**：不应直接采用。当前仓库中 `progressPercent` 不是派生字段；若直接改成只读自动计算，会改变现有产品语义。
- **“逾期三处联动修复，三处缺一不可”**：表述过强。详情、列表、卡片应有一致的时效感知，但不应强制把时效状态写入状态 pill。
- **“评论样式不一致”**：针对当前仓库不成立。现有评论卡片样式已统一，这条更适合留在静态稿评审，不应保留为当前产品结论。
- **“必须增加评论 / 活动 / 全部 Tab”**：不是当前最优先动作。当前已分区，首要问题是活动日志降噪，而不是再引入更重的筛选模型。

## 3. 当前产品语义约束

### 3.1 进度不是纯派生字段

- 当前 `Task.progressPercent` 是领域字段，编辑器侧栏可直接修改，并与 status 有联动。
- 因此后续优化必须避免把“子任务完成率”与“任务进度”直接混为同一概念。
- 更合理的方向是增强解释与展示层级，而不是立即移除手动进度。

### 3.2 评论与活动已分区

- 当前任务详情中，评论与活动已经是两个独立 section。
- 后续不应把问题定义为“系统日志与人工评论完全混排”，而应改写为“活动区自身缺少降噪与聚合能力”。

### 3.3 子任务区已具备基础能力

- 当前子任务区支持真实列表、状态修改、点击进入详情、嵌套显示与内联创建。
- 优化重点应是提升协作信息密度与状态可感知性，而不是重复提出“把占位文案改成真实列表”。

## 4. 推荐优化方案

### 4.1 第一阶段：可直接基于现有能力优化

1. **右侧属性面板分组**
   - 将属性重排为三组：执行（状态 / 优先级 / 负责人）、时间（计划开始 / 截止日期 / 进度）、归档（标签 / 项目）。
   - 目标是降低扫描成本，不改变数据模型。

2. **截止日期时态显式化**
   - 在详情侧栏为截止日期补充相对时间文案：`还有 X 天`、`今天截止`、`已逾期 X 天`。
   - 已逾期场景仅在 due date 本身与辅助文案上加强警示，不强制污染状态控件语义。

3. **附件空态折叠**
   - 无附件时只保留单行入口，例如 `附件 0` 或 `+ 添加附件`。
   - 有附件时再展开为列表，减少内容区无效高度。

4. **活动流降噪**
   - 同 actor、同字段、短时间内连续变更可前端合并显示。
   - 系统活动继续使用低强调样式，并统一 actor 姓名展示。

5. **翻页来源上下文补全**
   - 从列表上下文进入详情时，展示来源列表名，例如 `我的任务 · 10 / 31`。
   - 无来源上下文时维持当前展示或隐藏翻页。

6. **子任务状态语义强化**
   - 强化状态控件颜色与 hover / focus 提示，让用户更容易理解“这是状态控件，不是完成 checkbox”。

7. **评论输入区轻量增强**
   - 增加 `Cmd+Enter` 发送。
   - 优化“提醒成员”文案与 chip 的可操作提示。

### 4.2 第二阶段：需要先明确产品语义再推进

1. **进度字段语义重构**
   - 备选方案 A：保留手动进度，同时明确其与子任务完成率不同。
   - 备选方案 B：同时展示 `Progress %` 与 `Sub-issues x/y`，但建立主次层级。
   - 备选方案 C：彻底改成由子任务派生的只读进度；此方案需要单独产品决策。

2. **子任务协作信息扩展**
   - 在子任务行增加 assignee avatar、due date chip 等信息。
   - 推进前需先确认行高、缩进、多级场景横向空间，以及与列表视图策略的一致性。

3. **评论 / 活动筛选模型**
   - 是否增加 Tab 或筛选器，应在活动流降噪后再评估，不作为当前批次前置项。
