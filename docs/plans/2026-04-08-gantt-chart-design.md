# 甘特图视图设计说明

**日期**: 2026-04-08  
**状态**: 已定稿（基于讨论：可编辑、成熟开源组件优先、Frappe Gantt + Spike）

---

## 1. 背景与目标

- **目标**: 在项目工作区增加 **甘特图** 布局，与现有 Board / List 并列；用户在时间轴上 **拖拽改期** 并 **持久化** 到后端。
- **交互（已定）**:
  - 拖 **左/右沿**：分别只改 **计划开始** / **截止日期**（工期可变）。
  - 拖 **条中间**：**开始与截止同日平移**，**日历天数工期不变**。
- **技术取向**: 优先 **Frappe Gantt（MIT）** + 薄 Vue 封装；实现前 **Spike** 验证上述三种拖拽是否均满足；不满足则评估 `ddmy/vue3-gantt`（Apache-2.0）或许可方案（如 dhtmlx）。
- **数据**: 不新增甘特专用字段；使用现有 `Task.plannedStartDate`、`Task.dueDate`（毫秒时间戳）。

---

## 2. 方案选择（摘要）

| 方案 | 许可 | 取舍 |
|------|------|------|
| **Frappe Gantt + 封装**（首选） | MIT | 成熟、体小；需 Spike 确认拖拽语义 100% 匹配。 |
| **ddmy/vue3-gantt** | Apache-2.0 | Vue 原生；社区小于 Frappe，需实测交互与样式融合成本。 |
| **dhtmlx-gantt** | 商业 / GPL | 功能全；商业产品通常需付费许可，仅作备选。 |

---

## 3. 架构与数据流（已确认）

1. **入口**: 扩展 `ViewType` / `viewConfig.layout`，新增 **`gantt`**，与 `board`、`list` 同级切换（持久化规则与现有视图配置一致）。
2. **数据源**: 与 Board 共用 **`taskStore`** 中当前项目的任务集合；映射为甘特库所需结构（`id`、`start`、`end` 等，以 Spike 后 API 为准）。
3. **写回**: 拖拽 **结束** 时调用现有 **`updateTask(id, { plannedStartDate, dueDate })`**（或等价 PATCH）；**乐观更新** UI，失败则 **回滚条形** 并提示错误。
4. **后端**: 沿用既有任务更新接口，不新增聚合或甘特专用 API。

---

## 4. 缺省日期与可见性规则

| 情况 | 规则 |
|------|------|
| 同时有 `plannedStartDate` 与 `dueDate` | 正常绘制条；`end` 按「含结束日」语义与库约定对齐（若库为 exclusive end，在映射层 +1 日）。 |
| 仅有 `dueDate` | 绘制 **单日条**：开始与结束均为该日历日（与现有列表/详情日期展示同一时区策略，建议 **本地日历日** 从毫秒戳换算）。 |
| 仅有 `plannedStartDate` | 绘制 **单日条**（该日）；用户可通过拖右沿补出截止。 |
| 两者皆无 | **不绘制** 甘特条（避免编造虚假工期）；首版可选在图上方或空态文案中提示「N 项未设置日期」，**不阻塞** 发布。 |

**工期（天数）**: 由起止日历日推导；中间平移时保持「结束日 − 开始日」的 **日历天数** 不变（与用户对「工期不变」的表述一致；若遇夏令时/边界，以日历日差分为准）。

---

## 5. 子任务与筛选

- **首版（YAGNI）**: 甘特中 **仅展示顶层任务**（`parentId` 为空或未定义），与列表在「不展示子任务」时的心智一致。
- **后续迭代**: 可按 `showSubIssues` 与父子关系做缩进行或分组，不纳入本设计范围。

---

## 6. 时间刻度与国际化

- **首版**: 使用 Frappe 内置视图粒度（如 **日 / 周**）；与产品现有 **vue-i18n** 一致，库内固定英文月份名等若无法注入，可接受首版英文刻度或后续 CSS/配置覆盖。
- **周起始**: 若与全局设置冲突，以 **产品现有日期设置** 为准做一次对齐（实现阶段核对 `TaskRowInlineDateCell` 等同源逻辑）。

---

## 7. 错误处理与一致性

- **网络失败**: Toast + 从 `taskStore` 最新状态 **重绘** 该条或全表刷新（取实现成本较低者）。
- **并发**: 不引入复杂 OT；以 **服务端返回为准**，失败或收到冲突时可 `fetchTasks` 刷新当前项目列表。
- **提交频率**: 在 **指针释放 / 库 `on_date_change` 单次回调** 时 PATCH，避免拖拽过程高频请求。

---

## 8. 测试策略

- **单元测试**: 纯函数 `task ↔ 甘特行` 的日期映射（含仅截止、仅开始、双字段、边界日）。
- **Spike 验收**: 人工或录屏确认三种拖拽与写回字段正确。
- **E2E（可选）**: 若现有 E2E 框架成熟，可补一条「打开甘特 → 拖条 → 断言 API/列表日期」；非首版阻塞项。

---

## 9. 风险与缓解

- **Frappe 不完全支持目标交互**: Spike 早期验证；不通过则切换备选库或缩小首版交互范围（需产品确认，默认走备选库评估）。
- **仅单日任务与「工期不变」**: 单日条中间平移仍为单日（工期 0 天不变），行为自洽。
